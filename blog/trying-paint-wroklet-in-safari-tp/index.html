Trying Paint Worklet in Safari Technology Preview

As a big fan of CSS Houdini, I was so happy that Safari team decided to take Paint API in development. The first implementation was shipped with Safari Technology Preview (TP) 69. Initial setup

Before I was going to start my experiments, I decided to try demos I‚Äôve created before. They were working in Chrome, but some of them need Chrome Canary because some of the other Houdini parts are used there. So I‚Äôve navigated to the basic Paint Worklet demo‚Ä¶ and it was dead üò≠.

Dead demos inquiry

Instead of four circles, there was the white background on the screen. That means that fallback value hasn‚Äôt applied, because I‚Äôve used black color for it. That is interesting üßê. When I‚Äôve opened the inspector, I saw that background with paint function was applied to the element. Then I opened the console and there was an error ‚Äì ReferenceError: Can't find variable: paint. That was confusing, I haven‚Äôt used any variable called ‚Äúpaint‚Äù in the Worklet class definition. So I‚Äôve tried to load it manually in the console:

And got the same error again. Hm, addModule method should accept the path to the file with the custom painter implementation. There is a security-related restriction, similar to Web and Service Workers, that all Worklets code should be written in a separate file. They have completely different JavaScript context and shouldn‚Äôt have access to any global data. So I‚Äôve tried different path variation:

I‚Äôve got the error again, but a different one ‚Äì SyntaxError: Unterminated regular expression literal '/paint.js'. Seems that the function tried to interpret path as a code. There were links to the changed Safari source code in the release notes. So I‚Äôve decided to find an explanation there. Like Chrome, Safari wrote in C++. I‚Äôm not a big expert in this language, but at least can understand what is going on.

Source code Safari

I‚Äôve started exploration opening links to changes related to Paint API. The first thing I checked was tests as they are the best source of truth for the current implementation. They were just HTML files and looked like that (I‚Äôve made example shorter to show the main points):

<script id="code" type="text/worklet">
class MyPaint {
  paint(ctx, geom) {
    ctx.fillStyle = "red";
    ctx.fillRect(0, 0, geom.width, geom.height);
  }
}

<script type="text/javascript">
  importWorklet(
    CSS.paintWorklet,
    document.getElementById('code').textContent
  );

<style>

    background-image: paint(my-paint);

<div id="paint">div>

The Worklet code was stored in the script tag with custom mime type ‚Äútext/worklet‚Äù and then the text content was passed to the importWorklet helper. It confirmed my guesses and I continued with source code lookup. After some time I‚Äôve found the great comment in Worklet.cpp file:

// https://bugs.webkit.org/show_bug.cgi?id=191136

So I‚Äôve followed the link to the bug in WebKit Bugzilla. The comment explained everything:

Currently, a call to Worklet::addModule(String url) uses the url as code. It should fetch the script asynchronously, following the spec. ‚Äì Justin Michaud 2018-10-31 15:12:30 PDT

Instead of fetching the file addModule method parsed the code from a string ü§¶‚Äç‚ôÇÔ∏è.

Workaround

My finding meant that I need to transform my JS files with painters into a string, but for Safari only. Until they will implement the method according to the spec. There are a few possible solutions:

  1. Write code as a string üòÖ
  2. Write code inside script tag with a custom mime type ü§°
  3. Fetch JS files manually and transform to a string ü§†

I decided to go with the third option. I want to detect Safari browser, made a request for a script, and convert it to a string.

Detect Safari

In the current version of demos, I‚Äôve used feature detection, before registering Paint Worklet:

if ('paintWorklet' in CSS) {
  CSS.paintWorklet.addModule('paint.js');
}

After that, I need to detect Safari browser. After a short investigation, I‚Äôve used the next user agent check snippet:

if ('paintWorklet' in CSS) {
  if (navigator.userAgent.includes('Safari')
    && !navigator.userAgent.includes('Chrome')) {
    // Safari TP fix will be here.
  } else {
    CSS.paintWorklet.addModule('paint.js');
  }
}

This is slightly confusing, but Safari‚Äôs user agent should contain "Safari", but not "Chrome". Because Chrome has both of them ü§£.

Request the file

This is the easiest task, as Safari support most of the modern JavaScrip features. I used Fetch API to request the file. Then I parsed the response as Blob:

(async function() {
  const response = await fetch('paint.js');
  const blob = await response.blob();
})();

We are not allowed to use await outside asynchronous functions, that is why I wrapped the call in async IIFE.

Convert file to a string

The simplest way to transform Blob into a string, I‚Äôve found, was to use File Reader API. The semantics of this API looks different from modern ones as it is pretty old. First, we need to create FileReader instance. Then listen to the ‚Äúload‚Äù event. Only after that, we can start the reading process:

const reader = new FileReader();

reader.addEventListener('load', () => {
 CSS.paintWorklet.addModule(reader.result);
});

reader.readAsText(blob);

This was the final stage after that almost all demos worked in Safari üéâ. Some issues still there and I‚Äôm going to fix them in the nearest time. It was a fun journey, but as I‚Äôve found in the source code, Safari team implemented a lot of features that are not finished in Chrome Canary. But this is a completely different story and I will definitely share my experiments in the nearest future. Here is the resulting loading code:

if ('paintWorklet' in CSS) {
 // Safari TP fix.
 // Safari only accepts source code as a string instead file path.
 if (navigator.userAgent.includes('Safari')
   && !navigator.userAgent.includes('Chrome')) {
   (async function() {
     const response = await fetch('paint.js');
     const blob = await response.blob();
     const reader = new FileReader();

     reader.addEventListener('load', () => {
       CSS.paintWorklet.addModule(reader.result);
     });

     reader.readAsText(blob);
   })();
 } else {
   CSS.paintWorklet.addModule('paint.js');
 }
}

Conclusion

Making something undocumented and experimental is always a nice way to explore the Web platform APIs. And the most important it learns us to keep comments and tests up-to-date as they are the main source of truth. Have fun üëª!
